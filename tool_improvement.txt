Here's my assessment:

Tools with "adjudication/verdict" framing that should be renamed

\- validate-adjudication → rename to validate-evidence-pack or just validate

\- graph-build (docs say "from adjudication JSON") → rename to graph-from-evidence

\- report-from-evidence (docs: "Validate adjudication, build graph, and build report") → rename to build-report

The whole pipeline assumes you start by writing a formal adjudication JSON — but for exploration work like mine, that's backwards. I only know what's interesting after I investigate.

What was actually missing from the toolset

1\. No work-info tool — I couldn't look up X26n0534's title/author/period directly. I had to use source-read as a metadata workaround, which is heavy (reads actual text). A lightweight tool taking source\_work\_id → title, author, period, canon, traditions, description would save a roundtrip.

2\. No passage-summary or seed-brief tool — source-investigate returns raw data (frontier + similarity + vector neighbors), which is great, but I still had to manually synthesize everything into a report. A tool that takes a passage\_id and produces a structured research brief would directly serve the use case.

3\. frontier error was misleading — When I passed a work\_id instead of a passage\_id, it said "Passage not found" rather than "Expected a passage\_id, got a work\_id." That lost me time.

4\. report-build / pdf-build don't accept free-form input — I couldn't pipe my markdown into pdf-build because it expects adjudication JSON. A render-markdown flag or an input\_markdown parameter (like pdf-build already has!) would let me skip the adjudication middleman entirely.

Actually — pdf-build already has input\_markdown and input\_json params. That's exactly what I needed. I just didn't notice because the tool docs say "Use when producing a dossier/report artifact from completed evidence" which sounded like it required adjudication. If the doc said "Accepts raw markdown too" I would have used it.
